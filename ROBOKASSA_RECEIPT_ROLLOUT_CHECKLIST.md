# Robokassa Receipt Rollout Checklist

## 1) Pre-check env and routing
- Confirm `ROBO_ID`, `ROBO_SECRET1`, `ROBO_SECRET2` are set in production.
- Confirm payment links are generated by `index.js` (not legacy `server.js`).
- Confirm callback endpoint remains `POST /webhook/robokassa`.

## 2) Receipt payload contract
- Add `Receipt` JSON with:
  - `sno: "patent"`
  - `items` with exactly one item:
    - `name` (sanitized, max 128 chars)
    - `quantity: 1`
    - `sum` equal to `OutSum` numeric value
    - `tax: "none"`
    - `payment_method: "full_prepayment"`
    - `payment_object: "service"`

## 3) Signature rule (critical)
- Build two versions:
  - `receiptRawJson = JSON.stringify(receipt)` for signature
  - `receiptEncoded = encodeURIComponent(receiptRawJson)` for URL
- Sign only:
  - `MD5(MerchantLogin:OutSum:InvId:receiptRawJson:Password1)`
- Do not use encoded `Receipt` in the signature string.

## 4) OutSum normalization
- Format `OutSum` as string `0.00`.
- Use this same value as source for `Receipt.items[0].sum`.
- Ensure no mismatch between operation amount and receipt amount.

## 5) Service name priority
- Resolve item name in this order:
  1. `tariffLabel`
  2. `courseName`
  3. `studioName`
  4. fallback: `Тренировка по калистенике`
- Sanitize and trim before adding to `Receipt`.

## 6) Regression guard (callback)
- Keep ResultURL verification unchanged:
  - `MD5(OutSum:InvId:Password2)`
- Keep existing paid-side effects unchanged (Airtable/YDB/Telegram branches).

## 7) First-payment validation
- Generate a real Robokassa link and verify URL contains `Receipt=`.
- Open payment page and verify no signature error (especially error 29).
- Execute one successful payment in test/limited scope.
- Confirm callback returns `OK{InvId}` and purchase status becomes `paid`.
- Confirm payer/admin notifications still arrive.
